# CLAUDE.md

本文件为 Claude Code（claude.ai/code）在本仓库中工作时提供指导。

## 代码生成

本 SDK 的大部分代码由内部 OpenAPI 代码生成器 `oagen` **自动生成**。生成的文件以 `# This file is auto-generated by oagen. Do not edit.` 开头。手动维护的代码使用 `@oagen-ignore-start` / `@oagen-ignore-end` 标记进行隔离，或标记为 `# @oagen-ignore-file`（例如 `session.py`、`passwordless.py`、`vault.py`、`public_client.py`、`pkce.py`、`actions.py`）。

## 开发命令

### 安装与设置

```bash
uv sync --locked --dev # 以开发模式安装包，并安装开发依赖
```

### 代码质量

```bash
uv run ruff format .          # 格式化代码
uv run ruff format --check .  # 检查格式化，不修改文件
uv run ruff check .           # 代码检查
uv run pyright                # 类型检查
```

### 测试

SDK 使用 [nox](https://nox.thea.codes/) 和 [nox-uv](https://github.com/dantebben/nox-uv) 进行多版本 Python 测试，确保兼容所有支持的 Python 版本（3.10-3.14）。

**使用测试脚本快速测试：**

```bash
./scripts/test.sh              # 在所有 Python 版本上运行测试
./scripts/test.sh 3.12         # 仅在 Python 3.12 上运行测试
./scripts/test.sh 3.11 3.12    # 在 Python 3.11 和 3.12 上运行测试
./scripts/test.sh --coverage   # 在所有版本上运行测试并生成覆盖率报告
./scripts/test.sh --ci         # 运行完整 CI 检查（检查、类型、测试）
./scripts/test.sh --fresh      # 重新创建虚拟环境
./scripts/test.sh 3.12 -- -k "test_sso" -v  # 传递 pytest 参数
```

**直接运行 nox 命令：**

```bash
uv run nox                       # 在所有 Python 版本上运行测试
uv run nox -s tests-3.12         # 在指定 Python 版本上运行测试
uv run nox -s coverage           # 运行测试并生成覆盖率报告
uv run nox -s lint               # 运行代码检查
uv run nox -s typecheck          # 运行类型检查
uv run nox -s ci                 # 运行所有 CI 检查
uv run nox -l                    # 列出所有可用会话
```

**单版本测试（开发时更快）：**

```bash
uv run pytest                    # 在当前 Python 版本上运行所有测试
uv run pytest tests/test_sso.py  # 运行指定测试文件
uv run pytest -k "test_name"     # 运行匹配名称模式的测试
uv run pytest --cov=workos       # 运行测试并生成覆盖率报告
```

## 架构概览

### 客户端架构

SDK 同时提供同步和异步客户端：

- `WorkOSClient`（同步）和 `AsyncWorkOSClient`（异步）是主要入口点（从 `workos/__init__.py` 导出）
- 两者均继承自 `_BaseWorkOSClient`（位于 `workos/_client.py`），负责配置、HTTP 传输、重试逻辑和错误映射
- 每个功能领域（SSO、Organizations 等）以 `@functools.cached_property` 的形式暴露在客户端上，实现懒加载
- HTTP 传输直接使用 `httpx`，没有单独的 HTTP 客户端抽象层

### 模块结构

每个功能模块遵循以下结构：

```
src/workos/{module_name}/
  __init__.py        # 从 _resource 和 models 重新导出
  _resource.py       # 同步和异步资源类（例如 SSO + AsyncSSO）
  models/
    __init__.py      # 重新导出所有模型类
    {model}.py       # 独立的 dataclass 模型文件
```

资源类接收 `WorkOSClient` 或 `AsyncWorkOSClient` 的客户端引用，并调用 `self._client.request()` 或 `self._client.request_page()` 处理分页端点。

### 类型系统

- 所有模型使用 `@dataclass(slots=True)` —— **不是** Pydantic
- 每个模型实现 `from_dict(cls, data) -> Self` 用于反序列化，以及 `to_dict() -> Dict` 用于序列化
- `workos/_types.py` 中的 `Deserializable` 协议定义了 `from_dict` 的约定
- `RequestOptions`（一个 `TypedDict`）允许针对每次调用覆盖 headers、timeout、retries 等
- 类型检查使用 **pyright**（配置在 `pyrightconfig.json` 中）

### 分页

- `workos/_pagination.py` 中的 `SyncPage[T]` 和 `AsyncPage[T]` 数据类表示分页结果
- 基于游标：使用 `before`/`after` 属性，并通过 `has_more()` 判断是否还有更多数据
- `auto_paging_iter()` 透明地获取后续页面

### 错误处理

所有异常位于 `workos/_errors.py` 中，并继承自 `WorkOSError`：

- `BadRequestError` (400)、`AuthenticationError` (401)、`AuthorizationError` (403)、`NotFoundError` (404)、`ConflictError` (409)、`UnprocessableEntityError` (422)、`RateLimitExceededError` (429)、`ServerError` (5xx)
- `ConfigurationError`、`WorkOSConnectionError`、`WorkOSTimeoutError` 用于非 HTTP 错误
- `STATUS_CODE_TO_ERROR` 字典将状态码映射到异常类

### 测试框架

- 使用 **pytest** 和 **pytest-httpx** 进行 HTTP 模拟（提供 `httpx_mock` fixture）
- 同步（`TestSSO`）和异步（`TestAsyncSSO`）变体分别使用独立的测试类
- 异步测试使用 `@pytest.mark.asyncio`
- JSON fixture 位于 `tests/fixtures/`，通过 `tests/generated_helpers.py` 中的 `load_fixture()` 加载
- 客户端 fixture `workos` 和 `async_workos` 定义在 `tests/conftest.py` 中

### 配置

- 支持环境变量：`WORKOS_API_KEY`、`WORKOS_CLIENT_ID`、`WORKOS_BASE_URL`、`WORKOS_REQUEST_TIMEOUT`
- 重试逻辑：指数退避加抖动，在 429/5xx 时重试，并遵循 `Retry-After` 响应头
- 默认超时：60 秒
